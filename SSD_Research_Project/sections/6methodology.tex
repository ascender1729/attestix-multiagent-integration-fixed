\section{Workflow}

\subsection{Problem Definition}

Contemporary multi-agent AI systems deployed in regulated sectors exhibit fundamental security deficiencies. Four formal requirements are identified:

\begin{itemize}
    \item \textbf{R1 (Identity):} $\forall a_i$, there exists a cryptographic binding $\sigma_i = \text{Sign}(sk_i, \text{id}_i)$ where $sk_i$ is a private key unique to $a_i$.
    \item \textbf{R2 (Delegation):} Agent $a_j$ may invoke resource $r$ only if it presents a capability-scoped UCAN token $T$ issued by an authorising principal, where $T$ cryptographically binds the capability $c_r$ and is verified at the point of invocation. Single-hop delegations are exercised in this PoC; multi-hop attenuation chains $\delta = (a_{\text{root}} \to \cdots \to a_j)$ are enumerated in Future Works.
    \item \textbf{R3 (Provenance):} $\forall \alpha_i$, there exists a log entry $\ell_i = (\text{id}_i, \alpha_i, d_i, o_i, h_{i-1})$ where $h_{i-1} = \text{SHA256}(\ell_{i-1})$, forming a tamper-evident chain.
    \item \textbf{R4 (Compliance):} $\forall a_i$ in a high-risk domain, there exists a compliance profile $\pi_i$ mapping $a_i$ to an EU AI Act Annex III risk category.
\end{itemize}

No existing framework satisfies all four requirements simultaneously.

\subsection{Shared AttestixManager Design}

A shared parameterised \texttt{AttestixManager} class eliminates code duplication across the four domain projects. Each domain instantiates it via a factory function with domain-specific configuration, reducing per-domain security code to a 9-line configuration shim:

\begin{lstlisting}[caption=Domain-specific client instantiation (Hospital)]
# Hospital/c_agents/attestix_client.py
from attestix_client import make_attestix_client

attestix_client = make_attestix_client(
    issuer_name="Attestix General Hospital",
    capabilities=["medical_diagnosis", "patient_intake"],
    credential_type="MedicalBoardCertification",
    risk_label="Annex III - Health",
    domain_label="Hospital"
)
\end{lstlisting}

This design ensures that a bug fix or feature addition to the shared client propagates to all four domains simultaneously. The shared client additionally exposes Article 10 (\texttt{record\_training\_data}), Article 11 (\texttt{record\_model\_lineage}), and Article 43 / Annex V (\texttt{issue\_declaration\_of\_conformity}) helpers under EU AI Act conformity, plus a one-shot \texttt{setup\_full\_compliance(agent\_id, purpose)} that exercises all four pillars in a single call from each \texttt{compliant\_main\_*.py} entry point.

\subsection{Identity Provisioning (Pillar 1)}

At system startup, each agent receives a unique identity via \texttt{provision\_identity()}. The \texttt{IdentityService} generates an Ed25519 keypair, derives a \texttt{did:key} DID, and returns a Universal Agent Identity Token (UAIT):

\begin{lstlisting}[caption=Identity provisioning output]
[Attestix Identity] Provisioning Official Identity for Patient Intake Node...
   -> DID:  did:key:z6MkhpCDcsThDeVFcbrXTaVjUKU4qvSi1jJ1GCYYQ91Saz93
   -> UAIT: attestix:a5cdd3e3c7194201
\end{lstlisting}

\subsection{EU AI Act Compliance Profiles (Pillar 2)}

Each agent receives a compliance profile via \texttt{setup\_compliance()}, mapping it to an EU AI Act Annex III risk category. The \texttt{check\_conformity()} gatekeeper queries the local compliance store before permitting execution. If no profile exists, execution is blocked:

\begin{lstlisting}[caption=Conformity gatekeeper blocking an unregistered agent]
[Attestix Compliance] Conformity Gatekeeper: Enterprise DB Node...
   -> BLOCKED. 'Enterprise DB Node' has no compliance profile.
PermissionError: Attestix Conformity Assessment Failed.
\end{lstlisting}

\subsection{UCAN Delegation Gates (Pillar 3)}

The UCAN delegation pattern enforces least privilege between agents. The orchestrator creates a capability-scoped token before each agent handoff, and the receiving agent verifies it before executing:

\begin{lstlisting}[caption=UCAN delegation and verification]
# Orchestrator creates delegation
db_token = attestix_client.delegate_capability(
    INTAKE_ID, DB_ID, "read_patient_record"
)
# Receiving agent verifies before executing
attestix_client.verify_token(db_token, "read_patient_record")
\end{lstlisting}

\texttt{verify\_token()} in the integration wrapper delegates to the upstream \texttt{DelegationService.verify\_delegation()} routine, which performs JWT signature verification (EdDSA over Ed25519), expiry check, revocation-list lookup, and recursive proof-chain validation with cycle detection. The wrapper additionally enforces that the returned capability set includes the required scope before returning; any check failure raises \texttt{PermissionError} and blocks execution.

\subsection{Merkle Provenance Chain (Pillar 4)}

Every agent action is hash-chained into the provenance log via \texttt{log\_action()}. The SHA-256 chain hash links each entry to its predecessor, forming a tamper-evident audit trail displayed at the end of each run.

\subsection{W3C Verifiable Credentials (Pillar 5)}

Senior agents receive role-specific W3C Verifiable Credentials via \texttt{issue\_credential()}, signed with \texttt{Ed25519Signature2020} and stored locally in \texttt{\textasciitilde/.attestix/credentials.json}.

\subsection{Blockchain Anchoring (Optional Extension)}
\label{sec:blockchain}

The \texttt{BlockchainService} computes the RFC 6962 Merkle root of all audit entries and anchors it to Base Sepolia via EAS. An EVM wallet is required; the service fails gracefully when not configured. Two integration issues were identified during testing and addressed: (i) a load-order race condition where \texttt{BlockchainService} initialised before \texttt{.env} was loaded was resolved in the integration code by moving \texttt{load\_dotenv()} to module scope ahead of agent imports; (ii) an insufficient default gas limit of 300k versus an actual EAS attest cost of approximately 451k was identified during repeated reverts on Base Sepolia. The pragmatic workaround applied in the integration code was to pass an explicit higher gas limit at the call site (600k with EIP-1559 fee fields), and the finding has been recorded for the upstream Attestix library to fold the increased default into a future release.

\subsection{Threat Model and Limitations}
\label{sec:limitations}

The current proof-of-concept demonstrates the integration approach and exercises the cryptographic primitives end-to-end. It operates under three assumptions that a production deployment would need to relax:

\begin{itemize}
    \item \textbf{Trusted orchestrator process.} All four domain pipelines instantiate a single \texttt{AttestixManager} singleton inside the orchestrator. The singleton mints UCAN delegations on behalf of issuing agents using a server-wide signing key. A compromise of the orchestrator process therefore implies a compromise of the delegation authority for the session. Production deployment requires per-agent key material and out-of-process delegation issuance.

    \item \textbf{Local JSON store.} Identities, credentials, compliance profiles, and provenance entries are persisted to \texttt{\textasciitilde/.attestix/} as JSON without encryption-at-rest, rotation, or retention bound. Healthcare and finance deployments require an audited storage backend with GDPR Article 17 erasure support and tamper-evident on-disk encryption.

    \item \textbf{Single-hop delegations.} The four domain pipelines exercise capability-scoped tokens between two principals at a time. UCAN's capability-attenuation property through multi-hop chains is not exercised in the current PoC and is enumerated as Future Work.
    \item \textbf{Prompt-injection mitigation (OWASP LLM01).} Every external input flowing into an LLM prompt is now wrapped in spotlighting delimiters by a shared helper at \texttt{shared/prompt\_safety.py}, with an accompanying system-prompt suffix instructing the model to treat the delimited content strictly as data and never as instructions. A regex layer additionally flags the canonical injection vocabulary (\textit{ignore previous instructions}, \textit{disregard the above}, \textit{reveal your system prompt}, embedded \texttt{<system>} tags or delimiter strings) and annotates the wrapped block with a possible-injection note before forwarding it to the LLM. The guard is applied at the four intake points (patient statement, ticker, supplier name, contract requirements) and at the points where web-sourced content (DuckDuckGo) and internal documents are appended. Three pytest cases in \texttt{tests/test\_attestix\_integration.py} regression-test the wrap, the detection vocabulary, and idempotence of the system-prompt hardening. This is a defence-in-depth measure: it raises the cost of the common attack vectors but does not guarantee zero exploitation.
    \item \textbf{Synthetic data and lawful basis.} All six domain database files are synthetic; no real PHI, PII, customer records, trading IP, or licensed legal data is processed by the PoC. Each mock JSON carries a \texttt{\_synthetic\_data\_notice} key making the synthetic status machine-readable. Production deployment with real data in any domain would require (i) a documented lawful basis under DPDP Act 2023 or GDPR; (ii) a Data Processing Agreement with the LLM inference provider (Standard Contractual Clauses or an equivalent cross-border transfer mechanism); and (iii) data-minimisation and pseudonymisation at intake.
\end{itemize}

These limitations delineate the scope of the experimental claims made by this report; they do not invalidate the integration approach.

\subsection{UCAN Delegation Results}

All 11 delegation chains were verified successfully across all four domains.

\begin{figure}[H]
\centering
\includegraphics[width=0.80\textwidth]{carbon_regulated_p2.png}
\caption{Hospital regulated run: UCAN delegation gates for all three handoffs (Phase 2 DB, Phase 3 CrewAI, Phase 4 CMO) showing capability verification, conformity gatekeeping, and Merkle provenance logging per agent.}
\label{fig:carbon_reg_p2}
\end{figure}

Each gate follows the same three-step pattern: (1) the orchestrator mints a short-lived UCAN token scoped to a single capability and passes it to the downstream agent; (2) the receiving agent calls \texttt{verify\_token()} which checks the JWT signature, the \texttt{att} capability field, and the expiry --- blocking execution if any check fails; (3) the conformity gatekeeper then calls \texttt{check\_conformity()} against the local compliance store, confirming an EU AI Act profile exists before allowing the agent to produce output. Only after both gates pass does the agent write a provenance entry. This means every inter-agent handoff produces three independent artefacts: a delegation receipt, a compliance confirmation, and a Merkle-chained audit log entry --- all stored locally in \texttt{\textasciitilde/.attestix/} with no external dependency.

Five of seven adversarial safety probes passed outright (drug-allergy avoidance in Hospital, volatility flagging and invalid-ticker hallucination in Finance, unknown-supplier hallucination in ESG, no-liability-licence flagging in Court). Two partial scores remain as follow-up work: the palm-oil deforestation probe (missed the specific keyword \emph{biodiversity}) and the illegal cash-only tax-evasion probe (mentioned \emph{illegal} but did not produce an explicit refusal).

The Finance, ESG, and Court aggregate scores below 0.80 are driven by a common pattern: the model produces semantically correct decisions (\emph{recommend buying}, \emph{approved with conditions}) but does not emit the exact enumerated keyword (\texttt{BUY}/\texttt{SELL}/\texttt{HOLD}, \texttt{APPROVE}/\texttt{REVIEW}/\texttt{REJECT}/\texttt{SUSPEND}) that the rubric expects. A final structured-output stage with a Pydantic schema or an output parser is the natural next step to lift these scores.
